＜＜オブジェクト指向の基本＞＞📦
クラスは、インスタンスを生成するための設計図
インスタンスが帰属するクラスは1つだけ（単一クラスに属する）
クラスには、あらかじめ実行可能な行動をメソッドとして定義しておく
プログラムは、メッセージパッシングによってメソッドを呼び出すことで処理が進む
※ メッセージパッシングとは💡
→ 「このオブジェクトにこの処理を依頼する」という形で、オブジェクトに対してメソッド呼び出しを行うこと。（呼び出し側は内部実装を知らなくていい）

＜＜現実世界との違い＞＞🌍
現実世界では、先にモノ（実体）が存在し、それを後から分類（クラス化）する
人や動物に「泣け」と命令しても、必ずその通りに動くわけではない

→ オブジェクト指向は
現実世界を忠実に再現するものではなく、プログラミングのための割り切った仕組み
として理解する必要がある。

＜＜プログラミングのための仕組みとしてのオブジェクト指向＞＞🛠
クラス・ポリモーフィズム・継承は、ソフトウェアの 保守性・再利用性を高めるための仕組み
クラス：
データ（変数）と処理（メソッド）をひとまとめにし、独立性の高い部品を作るための仕組み

ポリモーフィズム・継承：コードの重複を減らし、汎用性の高い部品を作るための仕組み
インスタンス：
実行時に、インスタンス変数をコンピュータのメモリ上に展開した実体

→ これらを使いこなすことで、変更に強く、再利用しやすいソフトウェア構造を作れる。

＜＜プログラミング言語の歴史と背景＞＞🕰
コンピュータは1940年代に登場。当時は Java や C、COBOL、FORTRAN、Python、PHP などは存在しなかった
初期は 機械語で1行ずつプログラムを書いていた
（16進数表現が中心で、人間にとって非常に扱いづらい）

最初の改善が アセンブリ言語
→ 機械語を人間が理解しやすい記号で表現したもの
→ ただし、命令を少し間違えるだけでプログラムが暴走する危険があった

その後、高級言語が発明された
→ 命令を1行ずつ書くのではなく、
より人間にわかりやすい抽象的な表現でプログラムを書けるようになった

※ 高級言語とは
→ 機械に近いか遠いかではなく、人間の思考に近い形で書ける度合いが高い言語

＜＜ソフトウェア危機と構造化プログラミング＞＞🚨
高級言語により生産性と品質は向上したが、コンピュータの普及と需要の増加がそれ以上のスピードで進んだ
1960年代後半、NATOで「将来、世界中の人がプログラマになっても需要に追いつかない」と指摘され、ソフトウェア危機と呼ばれる問題が提起された
この中で、オランダの学者 エドガー・ダイクストラが 構造化プログラミング を提唱
 構造化プログラミングの考え方 📐

正しく動作するプログラムを作るには、分かりやすい構造が重要
プログラムの制御構造を次の3つに限定する
順次進行：上から順に命令を実行
条件分岐：条件によって処理を分ける（if / case）
繰り返し：条件を満たす間処理を繰り返す（for / while）
サブルーチン（関数）の独立性を高める考え方も、保守性向上のために早い段階から工夫されていた
構造化プログラミングはGOTOレスプログラミングとも呼ばれる
GOTO：プログラム内の任意の場所へ無条件にジャンプする命令
多用すると処理の流れが追えなくなり、スパゲッティコードになりやすい
＜＜まとめ＞✍️
オブジェクト指向は突然生まれた概念ではなく、「どうすれば人間が複雑なソフトウェアを扱えるか」という歴史的な試行錯誤の延長線上にある。
保守性・再利用性・理解しやすさを高めるための考え方として、クラス・構造化・オブジェクト指向が段階的に発展してきたことを理解できた気がします。
